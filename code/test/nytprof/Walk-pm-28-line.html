206=head1 NAME
207
208Data::Walk - Traverse Perl data structures
209
210=head1 SYNOPSIS
211
212 use Data::Walk;
213 walk \&wanted, @items_to_walk;
214
215 use Data::Walk;
216 walkdepth \&wanted, @items_to_walk;
217
218 use Data::Walk;
219 walk { wanted => \&process, follow => 1 }, $self;
220
221=head1 DESCRIPTION
222
223The above synopsis bears an amazing similarity to File::Find(3pm)
224and this is not coincidental.
225
226Data::Walk(3pm) is for data what File::Find(3pm) is for files.
227You can use it for rolling your own serialization class, for displaying
228Perl data structures, for deep copying or comparing, for recursive
229deletion of data, or ...
230
231If you are impatient and already familiar with File::Find(3pm),
232you can skip the following documentation and proceed with
233L</"DIFFERENCES TO FILE::FIND">.
234
235=head1 FUNCTIONS
236
237The module exports two functions by default:
238
239=over 4
240
241=item B<walk>
242
243 walk \&wanted, @items;
244 walk \%options, @items;
245
246As the name suggests, the function traverses the items in the order
247they are given. For every object visited, it calls the &wanted
248subroutine. See L</"THE WANTED FUNCTION"> for details.
249
250=item B<walkdepth>
251
252 walkdepth \&wanted, @items;
253 walkdepth \%options, @items;
254
255Works exactly like C<walk()> but it first descends deeper into
256the structure, before visiting the nodes on the current level.
257If you want to delete visited nodes, then C<walkdepth()> is probably
258your friend.
259
260=back
261
262=head1 OPTIONS
263
264The first argument to C<walk()> and C<walkdepth()> is either a
265code reference to your &wanted function, or a hash reference
266describing the operations to be performed for each visited
267node.
268
269Here are the possible keys for the hash.
270
271=over 4
272
273=item B<wanted>
274
275The value should be a code reference. This code reference is
276described in L</"THE WANTED FUNCTION"> below.
277
278=item B<bydepth>
279
280Visits nodes on the current level of recursion only B<after>
281descending into subnotes. The entry point C<walkdepth()> is
282a shortcut for specifying C<{ bydepth =E<gt> 1 }>.
283
284=item B<preprocess>
285
286The value should be a code reference. This code reference is used
287to preprocess the current node $Data::Walk::container. Your
288preprocessing function is called before the loop that calls the
289C<wanted()> function. It is called with a list of member nodes
290and is expected to return such a list. The list will contain
291all sub-nodes, regardless of the value of the option I<follow>!
292The list is a shallow copy of the data contained in the original
293structure. You can therefore safely delete items in it, without
294affecting the original data.
295
296The behavior is identical for regular arrays and hashes, so you
297probably want to coerce the list passed as an argument into a hash
298then. The variable $Data::Walk::type will contain the string
299"HASH" if the currently inspected node is a hash.
300
301You can use the preprocessing function to sort the items
302contained or to filter out unwanted items. The order is also preserved
303for hashes!
304
305=item B<preprocess_hash>
306
307The value should be a code reference. The code is executed
308right after an eventual I<preprocess_hash> handler, but only
309if the current container is a hash. It is skipped for regular
310arrays.
311
312You will usually prefer a I<preprocess_hash> handler over a
313I<preprocess> handler if you only want to sort hash keys.
314
315=item B<postprocess>
316
317The value should be a code reference. It is invoked just before
318leaving the currently visited node. It is called in void context
319with no arguments. The variable $Data::Walk::container points
320to the currently visited node.
321
322=item B<follow>
323
324Causes cyclic references to be followed. Normally, the traversal
325will not descend into nodes that have already been visited. If
326you set the option I<follow> to a truth value, you can change this
327behavior. Unless you take additional measures, this will always
328imply an infinite loop!
329
330Please note that the &wanted function is also called for nodes
331that have already been visited! The effect of I<follow> is to
332suppress descending into subnodes.
333
334=back
335
336All other options are silently ignored.
337
338=head1 THE WANTED FUNCTION
339
340The &wanted function does whatever verifications you want on each
341item in the data structure. Note that despite its name, the &wanted
342function is a generic callback and does B<not> tell Data::Walk(3pm)
343if an item is "wanted" or not. In fact, its return value is
344ignored.
345
346The wanted function takes no arguments but rather does its work
347through a collection of variables:
348
349=over 4
350
351=item B<$_>
352
353The currently visited node. Think "file" in terms of File::Find(3pm)!
354
355=item B<$Data::Walk::container>
356
357The node containing the currently visited node, either a reference to
358a hash or an array. Think "directory" in terms of File::Find(3pm)!
359
360=item B<$Data::Walk::type>
361
362The base type of the object that $Data::Walk::container
363references. This is either "ARRAY" or "HASH" or the empty string for
364everything else.
365
366=item B<$Data::Walk::seen>
367
368For references, this will hold the number of times the currently
369visited node has been visited I<before>. The value is consequently
370set to 0 not 1 on the first visit. For non-references, the value
371is undefined.
372
373=item B<$Data::Walk::address>
374
375For references, this will hold the memory address it points to. It
376can be used as a unique identifier for the current node. For non-
377references, the value is undefined.
378
379=item B<$Data::Walk::depth>
380
381The depth of the current recursion.
382
383=item B<$Data::Walk::index>
384
385Holds the index of the current item in the container. Note that hashes
386and arrays are treated the same. Therefore, if the current container is
387a hash and B<$Data::Walk::index> is even then B<$_> is a hash key. If
388it is odd, then B<$_> is a hash value.
389
390Note that the root container is the array of items to search that you
391passed to the wanted function!
392
393This variable has been added in Data::Walk version 1.01.
394
395=back
396
397These variables should not be modified.
398
399=head1 DIFFERENCES TO FILE::FIND
400
401The API of Data::Walk(3pm) tries to mimic the API of File::Find(3pm)
402to a certain extent. If you are already familiar with File::Find(3pm)
403you will find it very easy to use Data::Walk(3pm). Even the
404documentation for Data::Walk(3pm) is in parts similar or identcal
405to that of File::Find(3pm).
406
407=head2 Analogies
408
409The equivalent of directories in File::Find(3pm) are the container
410data types in Data::Walk(3pm). Container data types are arrays
411(aka lists) and associative arrays (aka hashes). Files are equivalent
412to scalars. Wherever File::Find(3pm) passes lists of strings to functions,
413Data::Walk(3pm) passes lists of variables.
414
415=head2 Function Names
416
417Instead of C<find()> and C<finddepth()>, Data::Walk(3pm) uses
418C<walk()> and C<walkdepth()>, like the smart reader
419has already guessed after reading the L</"SYNOPSIS">.
420
421=head2 Variables
422
423The variable $Data::Walk::container is vaguely equivalent to
424$File::Find::dir. All other variables are specific to the
425corresponding module.
426
427=head2 Wanted Function
428
429Like its archetype from File::Find(3pm), the wanted function of
430Data::Walk(3pm) is called with $_ set to the currently inspected
431item.
432
433=head2 Options
434
435The option I<follow> has the effect that Data::Walk(3pm) also
436descends into nodes it has already visited. Unless you take
437extra measures, this will lead to an infinite loop!
438
439A number of options are not applicable to data traversion and
440are ignored by Data::Walk(3pm). Examples are I<follow_fast>,
441I<follow_skip>, I<no_chdir>, I<untaint>, I<untaint_pattern>, and
442I<untaint_skip>. To give truth the honor, all unrecognized options
443are skipped.
444

449=head2 Recurse To Maximum Depth
450
451If you want to stop the recursion at a certain level, do it as follows:
452
453 my $max_depth = 20;
454 sub not_too_deep {
455 if ($Data::Walk::depth > $max_depth) {
456 return ();
457 } else {
458 return @_;
459 }
460 }
461 sub do_something1 {
462 # Your code goes here.
463 }
464 walk { wanted => \&do_something, preprocess => \&not_too_deep };
